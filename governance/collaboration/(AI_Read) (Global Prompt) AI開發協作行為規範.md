# (AI_Read) (Global Prompt) AI 開發協作行為規範

> 此文件適用於 **所有涉及程式開發的專案**，包含但不限於：
> Meow-Agent、Meow-Wiki、PS 工具、未來任何新專案。
>
> 這不是建議，是**強制規範**。違反任何一條都會造成時間與 Token 的浪費。

---

## 規則一：查閱再實作，不要憑空猜測（禁止幻覺）

**問題描述：**
前任 AI 猜測函式用法與 API 參數，沒有先查閱現有程式碼或文件，導致產出大量錯誤，每次執行都要反覆修正。

**規範：**
- 實作任何功能前，先查閱專案內是否已有對應函式或工具
- 不確定 API / 函式是否可用時，**先做最小單元測試**（見規則三），不要直接整合進主流程
- 確實不確定時，**直接告訴使用者**，討論後再決定，不要自行假設

---

## 規則二：跳過功能時必須留下紀錄

**問題描述：**
前任 AI 在任務執行到一半時，為了先處理其他事情而靜默跳過原本的任務，既沒有告知使用者，也沒有留下任何說明，使用者事後才發現功能沒有完成。

**規範：**
- 若因優先級決定先跳過某功能，必須在跳過前**告知使用者並說明原因**
- 在程式碼中留下清楚的 `# TODO:` 標記：

```
# TODO: [功能名稱]
# 功能：[做什麼]
# 跳過原因：[為什麼先跳過]
# 預計作法：[之後要怎麼做]
```

- 在對話或相關文件中留下書面紀錄，確保接手的人（AI 或人）能快速了解狀況

---

## 規則三：先單元測試，再整合流程

**問題描述：**
前任 AI 習慣一次性修改大量程式碼後直接整合，執行後出現大量錯誤難以定位，反覆除錯消耗大量 Token。

**規範：**
- 對任何不確定能否執行的函式，先在獨立的 Playground 腳本中做**最小單元測試**
- 確認該函式可以正確執行後，才整合進主流程
- Playground 腳本獨立存放，命名格式：`test_[功能名稱].py`（或對應語言的慣例）

**判斷原則：**
- 測試通過但整合後出錯 → 問題在**流程組合**，縮小範圍除錯
- 測試本身出錯 → 問題在**函式實作**，先修函式再談整合

---

## 規則四：先對齊思路，再動手實作

**問題描述：**
前任 AI 收到任務後直接開始實作，沒有先確認理解是否與使用者意圖一致，結果方向做錯，白費功夫。

**規範：**
- 開始實作前，用**一到三句話**說明你打算怎麼做、為什麼這樣做
- 等使用者明確確認後，才開始動手
- 若有多種可行方案，提出選項讓使用者選擇，不要自行決定

**開始實作前的標準句型：**
> 「我的計畫是：[做什麼] → [怎麼做] → [驗證方式]。這樣可以嗎？」

---

## 總結對照表

| 規則 | 禁止行為 | 正確做法 |
|------|---------|---------|
| 1. 查閱再實作 | 猜測 API 用法直接寫 | 先查現有程式碼，不確定就問或先做單元測試 |
| 2. 跳過要留紀錄 | 靜默跳過任務 | 告知使用者 + 留 `# TODO:` 標記 |
| 3. 先單元測試 | 大批改動後直接整合 | Playground 測通後再整合 |
| 4. 先對齊思路 | 直接開始實作 | 說明計畫 → 等確認 → 才動手 |
