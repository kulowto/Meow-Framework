# (Prompt) 角色創建架構

> 文件類型: prompt  版本: v0.1  建立: 2026-05-03  最後更新: 2026-05-03

## 用途

每次新建一個 Agent 角色時，依照此文件的架構填寫，確保所有角色文件格式一致，可被 AI 正確讀取與繼承。

---

```markdown

領域: [文案 / 程式 / 美術]  角色類型: [執行 / 品檢 / 歸檔]
---

# [角色名稱]

## 一、角色定位

（一句話描述這個角色的職責與邊界）

## 二、主要任務

- 任務二
- 任務三

## 三、工作流程

### 3.1 輸入規格

- 來源角色：[上游角色名稱，若來自使用者則填「使用者」]
```

## 各 Section 填寫說明

### 一、角色定位

描述這個角色的「職責邊界」，讓 AI 知道什麼事該做、什麼事不該管。

> 好的範例：「負責將搜集到的原始資料整理成精簡要點，不進行潤稿或創作，僅歸納事實。」

### 二、主要任務

3～5 條具體任務，用動詞開頭，避免模糊描述。

### 三、工作流程

**3.1 輸入規格**：定義上游交付的格式，是這個角色的「介面合約」。接收格式不符時，角色應停下並告知使用者。

**3.2 執行步驟**：「已固定的工作流程」注入點。步驟越具體越好，確保不同 AI 執行結果一致。

**3.3 輸出規格**：

| 欄位 | 說明 |
|------|------|
| 目標受眾 | 決定語氣與詳細程度 |
| 輸出格式 | 決定結構（條列 / 文章 / 報告） |
| 輸出風格 | 決定語調（冷靜中性 vs. 有溫度的對話感） |
| 輸出媒介 | 決定交付方式 |

> ⚠️ 同一 Pipeline 中，上游角色的「目標受眾」必須對應下游角色的「來源角色」，確保介面對齊。

### 四、專家知識

「專家知識」注入點，填入此角色執行任務所需的背景知識。可以是：
- 方法論（例如：AIDA 文案框架）
- 核心概念（例如：影音腳本的節奏原則）
- 領域規則（例如：SEO 標題的字數限制）

### 五、檢驗標準

「檢驗標準」注入點，每條標準應可明確判斷「通過 / 不通過」，避免模糊描述。

> 好的範例：「文字稿不含書面連接詞『因此』、『然而』，確保口語化。」

### 六、已知易錯點

初始留空，由優化角色在 QA 閉環結束後填入，成為下一次啟動時的歷史教訓。

建議格式：

```
- [日期] [問題描述] → [根因] → [修正方式]
```

### 七、參照 References

選填。需要借助外部 Skill 輔助生成特定內容時，在這裡宣告。

```
- Skill: karpathy-guidelines — 撰寫程式邏輯時載入，確保不過度設計
- Skill: ui-ux-pro-max — 生成視覺說明時載入
```

---

## 三種內容注入點對照

| 內容類型 | 對應 Section | 說明 |
|----------|-------------|------|
| 專家的知識 | 四、專家知識 | 框架、方法論、核心概念 |
| 檢驗的標準 | 五、檢驗標準 | 審查清單，每條可判斷通過與否 |
| 已固定的工作流程 | 三、3.2 執行步驟 | 必做事項的有序清單 |

---

## 覆蓋優先級

```
角色文件 > 領域文件 > 全局文件
```

同一欄位在多份文件中衝突時，以最接近角色的文件為準。全局文件定義「預設行為」，角色文件負責「場景覆蓋」。

---

## 建立新角色的檢查清單

- [ ] 角色定位是一句話，且明確說明邊界
- [ ] 主要任務每條都有動詞開頭
- [ ] 3.1 有填來源角色與接收格式
- [ ] 3.3 四個輸出規格欄位都已填寫
- [ ] 四、專家知識有至少一條內容
- [ ] 五、檢驗標準有至少一條可判斷的標準
- [ ] 六、已知易錯點初始留空（不填預設內容）
- [ ] 確認上游的「目標受眾」對應本角色的「來源角色」
