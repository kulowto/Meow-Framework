# Meow-Framework 通用 AI 協作規則

> 本文件是通用規則層，供客戶的 Agent CLAUDE.md 以 `@include` 繼承。
> 客戶在自己的 CLAUDE.md 第一行加入：`@../[Name]-Framework/CLAUDE.md`

---

## 程式開發協作規則（強制）

所有涉及程式開發的任務，無論在哪個專案，都必須遵守以下四條規範：

1. **查閱再實作**：先找現有函式或文件，不確定就做最小單元測試或直接問，禁止猜測 API 用法
2. **跳過要留紀錄**：跳過任何功能前告知使用者，並在程式碼留 `# TODO:` 標記說明原因與預計作法
3. **先單元測試**：新函式先在 Playground 獨立測試通過，再整合進主流程
4. **先對齊思路**：開始實作前說明計畫（做什麼 / 為什麼），等使用者確認再動手

完整規範文件：`governance/collaboration/(AI_Read) (Global Prompt) AI開發協作行為規範.md`

## Karpathy 程式碼行為原則（程式任務自動套用）

以下四條原則在任何程式相關任務時**自動生效**：

| 原則 | 行為要求 |
|------|---------|
| 1. 先想再寫 | 明確陳述假設；遇到歧義先提出多種解讀讓使用者選，不自行假設；有更簡單的方案時主動說出來 |
| 2. 最小實作 | 只實作被要求的功能；不加推測性功能、不做過早抽象、不加未被要求的「彈性設計」 |
| 3. 精準修改 | 只改需要改的部分；不重構相鄰程式碼；發現既有 dead code 標記但不刪除 |
| 4. 目標驗證 | 把抽象需求轉為可驗證目標；多步驟工作列編號清單逐項確認後才繼續 |

## 安裝工具標準流程（MCP / Skill / Plugin）

當使用者提供連結或提及「安裝某個工具」時，**依序執行以下步驟，不可跳過**：

1. **安全性檢查**：確認來源是否為官方或知名可信來源；檢查星數、更新頻率、Issues 是否有異常
2. **說明工具用途**：白話說明核心功能；列出 2-3 個具體使用情境
3. **反問使用者需求**：詢問想解決什麼問題，不假設使用者已確定要裝
4. **提出替代方案比較**：以表格列出各方案優缺點，讓使用者做最終選擇

安裝完成後，立即將工具資訊更新至 CLAUDE.md 的已安裝工具表格。

---

## 不確定問題的處理流程

遇到**需求不明確、問題有歧義、或有多種可行方向**時，開始執行前必須依序：

1. 使用 `AskUserQuestion` 工具向使用者提問，釐清核心需求
2. 提出**至少三種解決方案**，每個方案說明：方案描述 / 優點 / 缺點 & 限制
3. 等待使用者選擇後，才開始執行

禁止在未釐清需求的情況下自行假設並直接執行。

## 角色創建與資料傳遞規範

- **新建角色**：依照七個 Section 骨架填寫 → `governance/roles/(Prompt) 角色創建架構.md`
- **新增領域**：依照領域骨架填寫 → `governance/roles/(Prompt) 領域文件架構.md`
- **資料傳遞**：Agent 之間使用五種資料契約（派工單 / 交付包 / 釐清請求 / 壓縮包 / 錯誤回報單）
  定義於 `governance/collaboration/(AI_Read) (Global Prompt) 全域角色格式與規範文件.md` §4

## 文件管理規則

新建或刪除工作目錄下的任何文件時，**必須立即更新對應的 `MENU.md`**：
- 新建 → 在對應分組下補一行（格式：`| 文件短名 | 使用情境（≤15字） | AI讀/Prompt |`）
- 刪除或改名 → 同步移除或修改對應行
